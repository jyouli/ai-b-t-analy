## 一、总览：整体结构

本交付物是 **「AI 投标」能力的 Web 前端子应用**（可独立运行，也可嵌入公司统一主工作台）。在浏览器中，它承担**选标、商务标（审标）、资信库、消息**等界面的展示与操作编排，经企业统一**业务网关**与投标相关后端、PaaS 文档能力等协同。

**整体关系可概括为**：

- **表现层**：基于主流 Web 技术栈，采用组件化界面与**按页分包**，保障首体验与可维护性。

- **状态与业务编排层**：在页面与接口之间，集中管理**登录与租户上下文、多页签、各业务线列表/详情与流程状态**等，避免逻辑散落在零散脚本中。

- **网络与集成层**：所有业务与平台能力经**统一请求通道**走网关；支持**与主产品一致的会话、语言、版本**等协议；对长耗时或流式能力预留**流式数据**通道。

- **集成层**：与**主应用**通过标准化微前端方案对接，可接收主产品注入的**容器、跨应用消息总线**等，实现「一处登录、多应用协同」的体验。

以下从**与业务最相关的两块**先展开：**统一登录/会话/跳转**（第二节）、**与对象/元数据平台可迁移对齐的能力**（第三节）；其后再说明预埋能力、产品范围、技术架构等。

## 二、登录弹窗、会话与页面跳转（专项）

本小节供汇报时讲清：**谁在什么情况下会弹出登录、登录成功后页面会怎样、掉线与主动退出各怎么处理**。表述以**行为**为主，不列举源码文件。

### 2.1 登录弹窗如何被呼起（入口一览）

| 类型 | 场景说明 |
| --- | --- |
| **用户主动** | 侧栏**未登录**时点击**头像/「登录」文案**（**收起/展开**两种侧栏布局下入口一致），主动打开登录弹窗。 |
| **用户点消息** | 未登录时点击**消息铃**，不打开消息侧栏，**先弹出登录**；登录成功后再走消息能力。 |
| **业务操作前校验** | 在**新建选标/新建审标、上传/替换投标文件、部分公告/图片类操作**等需账号的步骤，若未登录，会**先弹出登录**再继续（避免误操作浪费接口）。以上入口均通过**统一的「需登录则弹窗」**能力实现。 |
| **请求发现会话无效** | 业务接口返回**约定好的会话过期类结果码**或 **HTTP 401** 时：清理本地**会话与账号展示状态**，并弹出登录。与主站/foreground 的**会话判断规则**一致。 |
| **全站消息通道（可选）** | 应用内有一路**全站事件**：任意模块可**广播「打开登录」**；**布局根节点**统一监听并展示同一套登录弹窗。统一请求里会话失败时，也经这条通道**唤起弹窗**（与「请求无效」合并理解即可）。 |
| **浏览器全局接口（可选）** | 在入口将「是否已登录 / 打开登录」挂到**浏览器全局**，便于与**旧脚本、主应用、运营页**在极少数场景下保持一致（一般用户无感）。 |

**去重与防连弹**  
全站有**“弹窗是否已打开”** 的互斥位：若已打开，**重复**「打开登录」**不会**再叠一层；避免接口风暴时疯狂弹窗。

### 2.2 登录成功后的不同处理（为什么有时只关弹窗、有时整页刷新）

登录需经**租户中心**完成**账号/短信等校验**，再经**企业入口服务**把**令牌、企业信息、用户档案**等写入本端存储。此后分两类体验：

1. **当前正停留在「三栏主导航可覆盖」的地址上**（选标、商务标、资信及其子路径）
   - 登录成功后**仅关闭**登录弹窗，**不整页刷新**，**不改变**当前地址。
   - 同时：布局会按「已登录」走后续逻辑（见 2.3）。

2. **当前**不在**上述菜单范围内**（例如深链、业务详情、未匹配到菜单的地址等）
   - 登录成功后对页面做一次**整页刷新**，以**用新会话**重新加载当前地址下的内容，避免仍停留在**半访客、半已登录**的中间态。

**流程内分支**（不额外改地址栏，仍属同一次「登录关单」过程）：

- 租户侧若返回**多企业**可选，会先在弹窗内走**选企业**再落会话，落会话后的行为仍按上面 1/2 两条分流。

- 若企业策略要求**短信二次确认**等，会进入**补充验证**子步骤；通过后再按上面 1/2 分流。未通过则停留在**登录弹窗**内，不视为登录成功。

- 因风控等原因返回**需图形验证码**时，在**同一张**登录表单内**叠加验证码**，不新开页面。

### 2.3 登录成功后还会自动做哪些事（与「跳转」的关系）

- **主数据与权限类数据**：一旦全局状态变为「已登录」，会**拉取**与本应用各业务相关的**主数据/列表依赖数据**（如企业内用户等公用参照）。**此处不涉及用户可见的路由跳转**，属于**数据层就绪**。

- **多页签模式**：若租户中心返回的账号配置里包含**多页签开关**结果，会同步到**本地多页签开关**与**全窗口可读变量**，与设置里**多/单页签**表现一致。

- **企业门户地址不一致**（多 Pod / 多入口域名）时：在**非本地开发、非预发**等场景下，若发现**当前站点**并不是该企业账号的**主 Web 入口**，会在登录成功后**整跳转到该企业对应门户地址**并携带**授权码**等参数，由**平台统一落地**到正确环境——这是**强合规**的「跳走」，与 2.2 的「为刷新会话而整页刷新」不同。

- **消息/推送通道**：**已登录**后初始化**即时消息/推送**相关能力；**未登录**或**退登**时**释放**连接，**节省**终端与长连接资源。

### 2.4 未登录与地址栏（路由级「跳转」）

- 若用户**未登录**，且当前地址**不属于访客允许**的范围，**自动**在浏览器层**重定向**到**主业务落地页**（商务标列表那类页面），不保留**需强登录的深链**在地址栏。

- 与 **2.2** 不冲突：那是「**已经**在可访客区登录成功以后」是否刷新；本款是「**还**在禁访区时」由框架**先**把地址**拉回**到允许区。

### 2.5 主动退出与清理

- 用户通过侧栏**头像/账号区**的**登出**入口时：清掉**与登录相关的本地存储**与**全局里的账号与租户展示**，并**销毁**消息通道。

- 若**登出时**当前地址**不在三菜单范围内**，会**重定向**到**主业务落地页**，与 **2.4** 的「回到可理解的主入口」体验一致。

- 登出**不会**自动弹出登录，由用户决定何时再点「登录」。

### 2.6 与「多页签、访客、消息」的交叉（汇报时可一句话带）

- **多页签**打开多地址时，**未登录+非访客**的地址**仍会被整体策略**收束到**主落地**；登录成功后，若在**可菜单化**的地址，则**不强制**跳 URL，**只**关弹窗与刷新数据。

- **消息铃**在**未登录**时等于「先登录再聊消息」；在**已登录**时**不再**为消息去弹登录。

### 2.7 安全与产品口径（可对外讲）

- **鉴权**与**租户中心、业务网关**协议对齐；不在前端长期持有**可逆密钥**，密码等按与**主站一致**的加密与盐流程提交。

- 会话在**统一请求**里**集中判定**；失效后**清痕迹 + 弹登录**，与**后台会话失效**叙述一致。

- **「记住我」**类选项若被用户使用，在本地有**与主站同套路**的存储行为（**含明文风险**的遗留约定），**安全审计**时应对齐主产品说明。

## 三、对象平台能力迁移支持范围

本应用内大量**业务列表、详情、新建/编辑**与集团**对象/元数据平台**（业务对象、字段与布局配置）配套使用。为便于**从主产品或平台侧迁移、复用同一套对象定义**，当前前端对以下能力有**明确定义的迁移/对齐支持范围**；未列项需按迭代单独评估。

### 3.1 字段类型（已支持）

以下字段类型在**列表、详情、表单**等形态中，按平台元数据驱动渲染与校验时，纳入本应用**约定支持**范围：

| 类型 | 说明（汇报用语） |
| --- | --- |
| **文本** | 常规单行/多行文本类展示与录入 |
| **单选** | 枚举/字典式单选 |
| **日期** | 日期或日期时间类 |
| **图片** | 图片上传、展示与平台存储衔接 |
| **查找关系** | 关联到其他业务对象的引用，与平台查找、回填规则一致 |
| **整数** | 整型数值 |
| **货币** | 金额类，与**高精度/展示格式**策略一致（见**第十四节**中「数值与计算」） |

> 未在上表中的字段类型，不代表永远不能支持，而表示**需单独排期**做**产品 + 元数据 + 联调**确认后再纳入基线说明。

### 3.2 列表页（已支持能力）

- **列宽拖拽**：列宽可交互调整，适配不同角色阅读习惯。

- **表头排序**：按列排序，与元数据/接口约定一致。

- **表头筛选**：表头区筛选，含常用类型（文本、选择、时间区间等，与**字段与字典**配置挂钩）。

- **列锁定**：左右固定关键列，便于宽表横滑时保持主键/名称列可见。

- **列表页布局**：列表区整体布局在**元数据/布局**约束下**可配置、可复用**。

### 3.3 详情页（已支持能力）

- **详情页布局**：详情区块、分组、只读区与平台**详情布局**能力对齐，**按对象**配置驱动展示。

### 3.4 新建与编辑（已支持能力）

在**新建/编辑**表单中，对以下**与平台行为一致**的逻辑能力有迁移支持，便于**规则下沉到配置**、减少硬编码：

- **默认值**：字段级默认取值。

- **公式**：与平台**表达式/公式**体系挂钩的计算与回填（在依赖链与性能允许范围内）。

- **条件显示**：随其他字段或上下文变化，**显隐**表单项。

- **条件必填**：在部分条件下动态**必填**校验。

- **条件只读**：在部分条件下**只读**、禁止修改。

**口径**  
上述能力以**对象平台当前版本与租户配置**为真源，前端负责**按元数据**还原交互；若平台升级引入新算子/新条件类型，需按版本做**联合回归**后更新本说明。

## 四、已具备的「预埋」与扩展能力（总览）

以下能力在架构上**已落地**，用于降低后续与主产品、多语言、运维策略对接时的返工。汇报时可概括为：在**主集成、国际化、多页签、多环境、长连接类场景**上已做系统化预埋。

| 能力域 | 预埋要点 | 价值说明 |
| --- | --- | --- |
| **微前端** | 既可独立访问，也可作为子应用被主工作台加载；可接收主产品传入的**挂载容器**与**跨应用消息** | 不绑定单一入口形态，与集团化「统一门户 + 多子应用」路线一致 |
| **统一请求与网关** | 业务请求**统一经网关前缀**与后端约定；自动携带**版本、语言、身份**等与平台协议对齐的头信息 | 与主站/foreground 的会话与**语言**策略一致，减少联调差异 |
| **多语言** | 文案**中英双语资源**可维护；**接口语言**与**界面语言**在登录/租户等场景下按平台惯例区分处理 | 满足出海或双语运营时「改文案、不改业务骨架」的迭代方式 |
| **多页签** | 用户可在设置中切换**多页签 / 单页签**；多页签下对**主入口页签固定、详情类页签可开可关**有完整策略 | 贴近办公场景下「多任务并行」习惯，与主产品多窗口体验可对齐 |
| **登录、访客与菜单** | 未登录时，对**三栏主导航**等路径开放为「访客可浏览」；需登录的深层能力在登录前**不抢占接口**，避免无效请求与体验抖动 | 兼顾获客/演示与正式使用的边界 |
| **多环境构建** | 工程支持**开发 / 测试 / 生产**等模式构建，**环境相关配置**与代码分离，由部署与流水线注入 | 满足测试、预发、生产隔离的行业常规要求 |
| **工程与构建韧性** | 对依赖库中非常规包形态、历史 `.js` 与 JSX 混用等，在构建层做了**兼容与转译** | 控制技术债对交付节奏的影响，保障可持续迭代 |

## 五、产品范围与使用形态

- **产品形态**：仅 **Web 浏览器**（不承载独立 App 壳），与产品「独立投标产品、仅 Web」方向一致。

- **三大主导航（用户可见）**
  - **选标**：招标要求摘录、与资信能力联动、分析结论与**报告式产出**等（以产品迭代为准）。
  - **商务标（审标）**：审标**历史、分析结果、检查项**等，默认作为进入本应用后的**主落地**之一。
  - **资信库**：企业资信**录入、查询、与选标/审标**的数据协同。

- **附属能力**：**消息中心**、通用**业务对象详情**等，支撑通知与跨模块跳转。

- **运行形态**
  - **独立运行**：可单独作为微站点部署与访问，便于专网或独立交付场景。
  - **嵌入主工作台**：以微前端子应用形态被主产品加载，共享登录态，并通过消息机制与主产品协同。

## 六、技术架构要点（非开发视角表述）

- **基座与界面**：采用业界主流的 **Vite 构建 + React 界面 + Ant Design 企业级组件**，并做**全站主题与品牌色**统一，与投标产品线视觉一致。

- **路由与访问结构**：使用**浏览器历史路由**；各业务模块**按需加载**（进入某功能时才拉取对应脚本包），控制首屏与后续导航体验。

- **全局状态**：采用成熟的数据流模式，将**账户与租户、多页签、各业务线数据**分模块管理，便于多人协作与问题定位。

- **与平台 PaaS 的衔接**：在**文件上传、文档预览（如 WPS 只读态）**等能力上，与 PaaS 及对象元数据体系对齐，**不重复造平台能力**。

## 七、路由与信息架构

从**用户能感知的路径**上，可概括为（具体以发版为准）：

| 能力 | 路径形态 | 体验说明 |
| --- | --- | --- |
| 选标列表 | 选标根路径 | 列表、检索、进入分析等 |
| 选标详情 | 选标下带**记录编码**的路径 | 多为**全屏/无侧栏**，聚焦阅读与分析 |
| 审标/商务标列表 | 审标根路径 | 历史记录、进入审标流程 |
| 审标结果/分析 | 审标下带**记录编码**的路径 | 全屏展示，避免侧栏干扰 |
| 资信库 | 独立根路径 | 与侧栏「资信库」文字对应 |
| 业务详情 | 通用详情路径 | 复用**业务对象**展示框架 |
| 消息中心 | 独立路径 | 与消息推送能力结合 |
| 未匹配地址 | 统一「未找到」页 | 与访客策略、登录策略配合 |

**布局策略**：需要沉浸阅读的页面可配置为**隐藏侧栏、全内容区**；标准列表与一级功能保留**可折叠侧栏**与**统一页头/面包屑**习惯。

## 八、与后端、网关的协作方式（请求与数据）

- **统一出口**：业务请求不直连零散域名，而经企业约定的**业务网关**（与后台设计中「与 PaaS、工程服务路径可区分」的叙述一致）统一进入投标及相关服务。

- **成功与失败语义**：与平台其他 Web 产品一致，通过**业务结果码、HTTP 状态、会话是否失效**等组合判断，**会话失效**时自动清理本地登录痕迹并**引导重新登录**，避免用户误以为已登录。

- **语言与平台偏好**：请求中携带**用户/企业语言**相关信息，**登录、租户、纯业务**等场景在语言策略上**与主产品对齐**，为后续**双语/多语运营**打基础。

- **拥塞与体验**：对并发请求有**限流与闲时补位**策略，对**大模型/流式**类接口走**流式数据通道**并支持**取消**，减少长时间卡死。

- **与投标服务前缀的对应**：设计文档中「投标 API 在统一网关下与既有路径区隔」的表述，在浏览器侧落实为**统一经网关的访问路径**；**具体接口以联调与接口管理平台为准**。

## 九、微前端与主产品集成

- **子应用身份**：在集团微前端体系中有**唯一应用名**，与主产品注册名一致，由主产品负责装配。

- **加载与隔离**：子应用有标准**安装、挂载、卸载**生命周期；卸载时释放在本页内创建的界面根节点，**避免切页后内存与事件堆积**。

- **与主产品通信**：支持主产品注入**跨应用消息总线**，本应用可在登录提示、多页签等场景与主产品**同构协同**；独立运行时则使用**内置消息机制**，行为保持一致。

- **跨应用上下文**：为兼容「子应用与主产品路由/调度器不完全相同」的情况，对**主产品下发的调度、历史**等预留**上下文透传**能力，减少重复接入成本。

- **跨域与资源**：在开发/联调场景下，对资源访问与跨域有**工程级策略**；生产环境以**同源与网关**策略为准。

## 十、多页签模式

- **可配置**：用户可在**设置**中在**多页签 / 单页签**之间切换。

- **多页签行为**
  - 保留一个与**默认主业务**绑定的**固定主标签**，不可关闭，保证用户始终有「家」可回。
  - 在**选标、审标**等存在「记录级详情」的功能上，为每条打开的分析/详情**自动或按需生成**对应标签，标题上体现**记录标识**便于区分。
  - 与**侧栏三大菜单**对应的路径，在打开时**与标签一一对应**，避免混乱。

- **与登录的配合**：在**未登录**且**不在访客允许范围**时，**不提前渲染**需强登录的页面内容，**避免无效接口调用**与闪烁；登录成功后的**地址是否变化**见**第二节**。

## 十一、访客与主流程权限

- **访客**可浏览的范围与**侧栏三菜单（选标、商务标、资信）及其子路径**对齐；未落在上述范围内的地址可能进入「未找到」页，但仍可按访客策略单独处理。

- **需登录**才能使用的功能（如新建、上传、强依赖账号的数据页），在**未登录**且**非访客区**时，会**重定向**到本应用约定的**主业务落地页**（与商务标列表一致），避免半屏白屏与无效数据请求。

- 与后台《AI 投标》中「**使用投标的用户可不按细粒度权限做模块划分**」的产品设定一致时，前端**不在本应用内做复杂模块级权限**（若产品策略变化，以产品为准再迭代）。

## 十二、多语言与地域化

- **界面文案**：维护**中文、英文**两套文案资源，通过统一函数取词，**支持占位符**（如人名、数量），便于**批量换文案**而不改逻辑。

- **接口侧语言**：与主站**个人/企业语言偏好**的存储键一致，**登录与租户**相关链路与**纯业务**链路在**传参策略**上区分，**与主产品行为对齐**。

- **当前界面语言**：**组件库**默认以**中文**呈现；**若未来**需与界面语言**完全同步切换英文组件库**，可在不改动业务结构的前提下，通过**配置层**完成升级。

- **工程化**：具备**从代码扫描、翻译表、到回写资源**的流水能力（供翻译协作与质量管控用），不强制阅读本文档的受众掌握细节，仅需知「**已具备可规模化多语交付的工程链**」。

## 十三、工程构建与多环境

- **构建目标**：产物面向现代浏览器，**ES 版本**有明确基线，平衡兼容与体积。

- **多环境**：通过**不同构建模式**区分开发、测试、生产等，**敏感地址与目标后端**不硬编码在业务逻辑中，**由环境配置与发布管道**注入。

- **子路径部署**：若主产品以**子路径**方式挂载子应用，需**门户、Nginx/网关与主产品配置**与前端**根路径**策略一致，属**联调/运维协同项**（本文不约定具体现网路径名）。

- **包体与依赖**：对依赖库中**非标准包形态、历史技术栈**在构建时做了**转译与兼容层**，**降低**因第三方包差异导致的**构建失败**概率。

## 十四、主要业务能力（用户可感知）

| 域 | 职责摘要 |

## 十五、非功能：安全、性能与可运维

- **安全与合规**：不将**密钥、账号**写死在仓库；生产配置由**安全渠道**注入；**HTTPS** 与**同源/网关**策略为默认假设；**Token** 存取方式与现网主站**一致**；富文本/ Markdown 等展示**遵循 XSS 防护**实践。

- **性能思想**：**按路由切包**、长列表在合适场景**虚拟滚动**、请求层**防短时洪峰**、大文件与流式**可取消**等。

- **可观测与协同**：对请求失败有**显式处理**；若后台要求**全链路 trace**，可在请求头**统一**挂接（**以后台与运维规范为准**）；生产环境**监控与日志**可复用**主产品已接入**的前端方案（若子应用与主产品同源部署，更易一体化）。
